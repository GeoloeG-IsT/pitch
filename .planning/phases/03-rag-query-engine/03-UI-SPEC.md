## Typography

Consistent with Phase 2 document page patterns (`text-[28px]`, `text-xl`, body). Two weights only: 400 (regular) for body-level text and 600 (semibold) for headings.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 16px | 400 (regular) | 1.5 | Streamed answer text, input placeholder |
| Label | 14px | 400 (regular) | 1.4 | Citation tags, status labels, section metadata |
| Heading | 20px | 600 (semibold) | 1.2 | "Sources" section heading, subheadings |
| Display | 28px | 600 (semibold) | 1.2 | Page title ("Ask a Question") |

Label is differentiated from Body by its smaller size (14px vs 16px), not by weight. This keeps the weight palette to exactly two values and avoids visual noise.

Source: Existing patterns from `apps/web/app/documents/page.tsx` (text-[28px] font-semibold, text-xl font-semibold)

## Color

Uses existing shadcn neutral theme tokens from `globals.css`. No new colors introduced.

| Role | Value | Usage |
|------|-------|-------|

| Secondary (30%) | `var(--color-card)` / `var(--color-muted)` | Citation card backgrounds, input field background, sources section |

Accent reserved for:
- "Ask Question" submit button (primary action)
- Text input focus ring
- Streaming cursor/caret indicator

## Component Inventory

### Primary Focal Point

**QueryInput** is the primary focal point of this page. It is the first interactive element the user encounters and the single entry point for all RAG queries. It sits at the top of the content area, visually anchored by the Display-sized page title above it and the answer area below it. The submit button within QueryInput uses accent color to draw attention.

## Interaction States

### QueryInput States

| State | Visual | Behavior |
|-------|--------|----------|
| Empty | Placeholder text visible, submit button disabled (muted) | No submission possible |
| Has text | Placeholder hidden, submit button enabled (primary) | Enter key or click submits |
| Submitting | Input disabled, button shows loading spinner | WebSocket connection establishing |
| Streaming active | Input disabled, button disabled | Cannot submit new question while streaming |
| Done | Input re-enabled, cleared, button enabled | Ready for next question |

### StreamingAnswer States

| State | Visual | Behavior |
|-------|--------|----------|
| Idle | Empty, hidden or showing previous answer | No activity |
| Retrieving | Skeleton pulse animation (3 lines) + "Searching documents..." label | WebSocket status message received |
| Generating | Tokens appearing character by character with blinking cursor at end | Tokens streamed from WebSocket |
| Done | Full answer displayed, cursor removed, Sources section appears | Citations message received, WebSocket closed |
| Error | Error message in destructive color with retry suggestion | Error message received or WebSocket closed unexpectedly |

### Streaming Cursor

- Blinking vertical bar (`|`) at the end of the last token
- CSS animation: `opacity` toggle at 530ms interval
- Removed when `done` message received

## WebSocket UX Contract

### Connection Lifecycle (user-visible)

| Phase | Duration | Visual Feedback |
|-------|----------|-----------------|
| POST /api/v1/query | <500ms typical | Button shows spinner, input disabled |
| WS connecting | <200ms typical | No visible change (covered by spinner) |
| WS status: "retrieving" | 1-3s typical | Skeleton animation + "Searching documents..." |
| WS status: "generating" | 2-10s typical | Tokens stream in, blinking cursor |
| WS citations | instant | Sources section populated |
| WS done | instant | Cursor removed, input re-enabled |
| WS error | instant | Error message displayed, input re-enabled |

### Reconnection Policy

No automatic reconnection in Phase 3. If the WebSocket closes unexpectedly:
- Display error state
- Re-enable input for retry
- User submits a new question to create a fresh connection

### Direct Backend Connection

Frontend WebSocket connects directly to FastAPI backend (not through Next.js proxy).
- Dev: `ws://localhost:8000/api/v1/query/{id}/stream`
- Configured via `NEXT_PUBLIC_WS_URL` environment variable
- Default fallback: `ws://localhost:8000`

Source: RESEARCH.md anti-pattern warning about Next.js WebSocket proxy limitation
